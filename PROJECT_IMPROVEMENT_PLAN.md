# План улучшения и оптимизации проекта volpt_edu

## 1) Текущая цель
Сделать проект быстрее, стабильнее и удобнее в поддержке без переписывания с нуля.

## 2) Базовые метрики (зафиксировать в первую неделю)
Перед оптимизациями нужно получить «точку А»:
- **Web Vitals**: LCP, INP, CLS (через Lighthouse + `useReportWebVitals`).
- **Размер JS-бандлов** по ключевым маршрутам (`/`, `/journal`, `/groups`, `/schedule`).
- **Время сборки** (`npm run build`) и наличие ESLint/TypeScript предупреждений.
- **Базовая стабильность UI**: ручной smoke-check основных страниц.

Критерии успеха (ориентиры):
- LCP < 2.5s на 75-м перцентиле.
- CLS < 0.1.
- Снижение JS на критических страницах минимум на 20%.
- Нулевые ошибки линтера в CI.

## 3) План работ по этапам

### Этап A — «Быстрые победы» (1–2 дня)
1. Подключить `next/image` для тяжелых изображений и SVG-иконок там, где это актуально.
2. Проверить лишние клиентские компоненты (`"use client"`) и оставить их только где нужен интерактив.
3. Убрать неиспользуемые зависимости/импорты, включить строгие правила ESLint.
4. Вынести повторяющиеся UI-шаблоны в переиспользуемые компоненты.

Ожидаемый эффект: уменьшение бандла и более быстрая первая отрисовка.

### Этап B — Оптимизация рендеринга и состояния (2–4 дня)
1. Для тяжелых списков/таблиц (`journal`, `groups`, `schedule`) применить:
   - `React.memo` для карточек/строк,
   - `useMemo/useCallback` на вычисления и обработчики,
   - стабильные `key` и минимизацию проп-дриллинга.
2. Перенести вычисления дат и фильтрации из рендера в хелперы/хуки с кэшированием.
3. Добавить lazy loading (`dynamic import`) для нефокусных блоков.

Ожидаемый эффект: меньше лишних перерисовок, плавнее интерфейс.

### Этап C — Архитектура данных и DX (3–5 дней)
1. Ввести единый слой доступа к данным (API-клиент/сервисный слой).
2. Типизировать DTO и ответы API (единые контракты в `types/`).
3. Добавить централизованную обработку ошибок и пустых состояний.
4. Настроить базовые e2e/smoke тесты (например, Playwright) для критичных сценариев.

Ожидаемый эффект: меньше регрессий, проще поддержка и масштабирование.

### Этап D — Наблюдаемость и контроль качества (1–2 дня)
1. Добавить проверку производительности в CI:
   - `npm run lint`,
   - `npm run build`,
   - Lighthouse CI для ключевых страниц.
2. Включить мониторинг рантайм-ошибок (например, Sentry).
3. Создать чеклист релиза: perf, accessibility, визуальная проверка.

Ожидаемый эффект: проблемы выявляются до продакшена.

## 4) Рекомендуемый технический бэклог
- Включить более строгий TypeScript (`strict`, `noUncheckedIndexedAccess`).
- Добавить алиасы импортов и разделить UI/domain/shared слои.
- Перейти к системному подходу в стилях (токены, единая шкала отступов/типографики).
- Добавить Storybook для визуальной регрессии компонентов.

## 5) Приоритизация (что делать первым)
1. **Сначала**: метрики + быстрые победы (Этап A).
2. **Затем**: оптимизация рендеринга (Этап B), потому что она дает самый заметный UX-эффект.
3. **После**: архитектура и тесты (Этап C) для устойчивого развития.
4. **Постоянно**: контроль в CI и мониторинг (Этап D).

## 6) Формат выполнения (спринтами)
- Спринт 1: Этап A + часть Этапа B (самые тяжелые страницы).
- Спринт 2: завершение Этапа B + Этап C.
- Спринт 3: Этап D + финальная стабилизация.

## 7) Что получить на выходе
- Документ с baseline-метриками «до/после».
- Отчет по размеру бандлов и Web Vitals.
- CI-пайплайн с обязательными проверками качества.
- Снижение визуальных и функциональных регрессий в релизах.
